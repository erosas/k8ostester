What's this cluster for?

Pick a starting point — it sets sensible defaults for everything below, which you can then tune. The manifest updates live as you edit.

Identity

Names the CNPG Cluster and the namespace it deploys into.

PostgreSQL image

The container image CNPG runs. Defaults to the official CloudNativePG PostgreSQL build — point it at a mirror or your own registry if you need to. Browse the repo's tags to pick a version and confirm it's pullable.

Topology & durability

How many instances (1 primary, the rest replicas) and how a write is acknowledged.

Storage

Persistent volume size, per instance.

⚠ Grow-only. You can expand this later only if the storage class allows online expansion, and never shrink it — start modest; adding space is easy, reclaiming it means a rebuild.

Compute (per instance)

CPU and memory each Postgres instance requests. Match limits to requests to give the pod Guaranteed QoS — it won't be evicted under node pressure and gets a predictable slice; leave it off for Burstable pods that can use spare capacity but may be squeezed. Sized per replica, so the cluster's total is this × instances.

Connection pooling

A PgBouncer -rw pooler multiplexes many client connections onto a few Postgres backends, and the app points at the <name>-rw pooler service. Recommended on: as your app's pods scale out they can open more connections than Postgres allows (max_connections) and stall the database — the pooler caps that.

Backups & recovery

Continuous WAL archiving + scheduled base backups to object storage. This is what makes point-in-time recovery possible — without it, Restore (PITR) stays disabled and there's no way back from a bad delete or a corrupt page. Leave off only for throwaway clusters.

Application roles & rotation

Creates two login roles, app_a and app_b, that own the same data (both inherit the app role). One is active at a time; the console's Rotate refreshes the idle role's password and flips the active pointer, so credentials rotate with no downtime window. Your app reads its password from the active role's Secret (the console shows which). Leave off if you rotate some other way.

Observability

How metrics leave the cluster. A Prometheus PodMonitor lets a Prometheus Operator scrape CNPG's metrics; an OTEL endpoint additionally ships them to your collector. With a scrape configured you also get an adaptive Grafana dashboard and the goals below.

Each goal draws a red waterline on its dashboard panel and a matching Prometheus alert rule — one number drives both. Leave blank to skip.
